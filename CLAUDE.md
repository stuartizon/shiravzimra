# Shira V'Zimra — Claude Code Rules

## Project overview

Shira V'Zimra is a Next.js website that displays a book of Jewish liturgical music arranged for male voice choir. The site is essentially a PDF viewer: a single compiled PDF (`/public/dist/shiravzimra.pdf`) is served via a Swiper-based viewer, with individual pieces navigable by URL (e.g. `/lecha-dodi`). Pieces are organised into groups (Shabbat, Festivals, Rosh Hashana, Yom Kippur, Misc, Songs) and sections within each group.

## Tech stack

- **Framework:** Next.js 14 (TypeScript)
- **Styling:** Tailwind CSS + CSS Modules
- **PDF rendering:** react-pdf (pdfjs-dist)
- **Carousel/navigation:** Swiper
- **Testing:** Jest + React Testing Library
- **Deployment:** Vercel

## Development commands

```bash
yarn dev          # start dev server
yarn build        # production build (also runs next-sitemap and PDF merge)
yarn test         # run Jest test suite
yarn test:watch   # run Jest in watch mode
yarn lint         # run ESLint
```

## Architecture

```
data/             # piece/section/group definitions in TypeScript
  types.ts        # Piece, Section, Group interfaces
  index.ts        # exports allGroups, allSections, allPieces, allPiecesMap
  shabbat/        # one directory per group, each with an index.ts + section files
  festivals/
  ...
pages/
  index.tsx       # main PDF viewer page
  [chapter].tsx   # re-exports index; getStaticPaths generates routes for all pieces/sections
  404.tsx         # 404 page
components/       # one directory per component
scripts/          # Node scripts for building the compiled PDF
  merge-scores.js # merges individual score PDFs + generates ToC into /public/dist/shiravzimra.pdf
public/
  dist/           # compiled shiravzimra.pdf (generated by yarn build)
  scores/         # individual piece PDFs, named by piece id (e.g. lecha-dodi.pdf)
```

## Adding a new piece

1. Add a PDF file to `public/scores/<id>.pdf`
2. Add a `Piece` entry to the relevant section file in `data/` with the matching `id`
3. Run `yarn build:scores-pdf` to regenerate the compiled PDF
4. The piece is now accessible at `/<id>`

## Coding conventions

- Functional components only, no class components
- One component per file; component directory matches component name
- CSS Modules for component-scoped styles, Tailwind for layout/utilities
- No default exports from utility or data files — named exports only
- All routes are generated via `getStaticPaths` in `[chapter].tsx`; do not add new page files

## Testing

Follow red-green-refactor TDD:
1. Write the tests for the new functionality first — they must fail before any implementation begins.
2. Implement the functionality.
3. Work is only complete when all tests pass.

Never mark a task done or move it to In Review until the test suite is green.

## Ticket workflow

This project is tracked on a monday.com board (board ID: `5097581269`).

When starting work on a ticket:
1. Move the item's Status to **In Progress** on the monday.com board.
2. Create a new git branch from the latest `master` — name it after a short descriptive slug, e.g. `alt-text-icons`.

When asked to create a PR:
1. Run `yarn lint` and `yarn test` locally and fix any failures before pushing.
2. Push the branch and open the PR against `master`.
3. Update the monday.com item's **PR Link** column (column ID: `link_mm3w3fym`) with the PR URL and a short label, e.g. `PR #3 alt-text-icons`.
4. Move the item's Status to **In Review**.

## Commit messages

Plain imperative sentences, no ticket numbers or scope prefixes. Example: `Add alt text to streaming icons`.
